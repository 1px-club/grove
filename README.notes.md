# Grove 实现补充说明

## 关键假设

- 用户与账户由上游系统创建，不在当前项目范围内
- 当前模型按“一用户一账户”设计，`accounts.user_id` 保持唯一
- 批量交易请求中，只要有任意一个 `userId` 没有对应账户，整批请求失败

## 设计取舍

### 金额表示取舍

- 对外仍使用十进制字符串，是为了让接口契约更贴近常见业务系统的金额表示
- 服务层和数据库内部改用最小货币单位整数，是为了把精度风险收敛在接口边界之外
- 返回时统一转换为两位小数字符串，是为了保证输入输出语义一致

### 账户与流水建模取舍

- 单独维护当前余额，是为了让查询余额保持简单直接，避免每次都从流水聚合
- 单独记录流水和 `endingBalance`，是为了保留交易后的确定性结果，方便核对与追踪
- 流水表冗余 `user_id`，是为了减少按用户维度查询时的额外关联成本
- 冗余字段继续交给数据库约束兜底，是为了避免便利性引入一致性风险

### 数据库变更策略取舍

- 关闭 `synchronize`，是为了避免应用启动时隐式改表
- 统一通过 migration 管理结构变更，是为了让数据库演进路径可追踪、可回滚
- 保留 [`init.sql`](./init.sql) 仅做示例数据准备，是为了把“建表”和“造数据”两个动作明确分开

## 并发策略取舍

- 选择数据库事务作为一致性边界，是因为余额与流水必须一起成功或一起失败
- 选择悲观写锁而不是把正确性留给应用层协调，是因为当前实现更优先保证结果确定性
- 在单个事务里按输入顺序逐笔处理，是为了让同批次结果具备稳定、可解释的业务语义
- 先完成余额计算，再统一提交余额更新和流水写入，是为了保持实现清晰且易于审查

## 错误语义

- `checkBalance = true` 时，如果任意一笔交易会导致负余额，整批请求失败
- 请求参数校验失败时，由全局 `ValidationPipe` 直接拦截
- 业务异常和通用异常都会被统一格式化为一致的错误响应结构

## 当前边界

- 当前对外入口是 `userId`，适用于“一用户一账户”场景
- 当前提供的是余额查询与批量交易能力，不包含流水查询接口
- 当前实现优先保证逻辑正确、数据一致和可读性
